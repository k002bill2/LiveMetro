# 앱 종합 검증 워크플로우

이 커맨드는 Boris Cherny가 강조하는 **검증 피드백 루프**를 구현합니다.
모든 코드 변경 후 이 커맨드를 실행하여 품질을 보장하세요.

## 검증 단계

### 1. TypeScript 타입 체크
```bash
npm run type-check
```
**통과 기준**: 타입 에러 0개

### 2. ESLint 린트 검사
```bash
npm run lint
```
**통과 기준**: 린트 에러 0개 (경고는 허용)

### 3. Jest 테스트 실행
```bash
npm test -- --coverage --coverageReporters="text-summary"
```
**통과 기준**:
- 모든 테스트 통과
- Statements: 75% 이상
- Functions: 70% 이상
- Branches: 60% 이상

### 4. 개발 빌드 검증
```bash
npm run build:development
```
**통과 기준**: 빌드 성공

## 결과 리포트 형식

검증 완료 후 다음 형식으로 결과를 요약하세요:

```
## 검증 결과 요약

| 항목 | 상태 | 세부사항 |
|------|------|----------|
| TypeScript | ✅/❌ | 에러 X개 |
| ESLint | ✅/❌ | 에러 X개, 경고 Y개 |
| 테스트 | ✅/❌ | X개 통과, Y개 실패 |
| 커버리지 | ✅/❌ | Stmt X%, Fn Y%, Br Z% |
| 빌드 | ✅/❌ | 성공/실패 |

**전체 상태**: ✅ 통과 / ❌ 실패
```

## 실패 시 조치

### TypeScript 에러
1. 에러 위치 확인
2. 타입 정의 수정 또는 추가
3. `any` 대신 `unknown` 사용

### ESLint 에러
1. 자동 수정 시도: `npm run lint -- --fix`
2. 수동 수정 필요 시 규칙 확인

### 테스트 실패
1. 실패한 테스트 확인
2. 예상값 vs 실제값 비교
3. 코드 또는 테스트 수정

### 빌드 실패
1. 빌드 로그 확인
2. 의존성 문제 확인: `npm install`
3. 캐시 정리: `npm run clean`

## 자동 수정 루프 (2-Strike Rule)

검증 실패 시 에이전트는 다음 규칙을 따릅니다:

1. **1차 수정 시도**: 실패 로그(전체 에러 메시지, 파일 경로, 줄번호)를 읽고 최소 침습적 수정을 시도
2. **전체 검증 재실행**: `/verify-app` 다시 실행 — 같은 단계가 또 실패하면 원인이 다른지 확인
3. **2차 수정 시도**: 1차와 다른 접근(예: 타입 수정 대신 타입 정의 변경, 테스트 수정 대신 구현 변경)
4. **STOP & REPORT**: 2회 연속 실패 시 즉시 중단하고 다음 형식으로 사용자에게 보고:
   ```
   ## 2-Strike 발동: 수동 개입 필요
   - 실패 단계: [TypeScript | ESLint | Test | Build]
   - 1차 시도: [무엇을 했는지]
   - 2차 시도: [무엇을 했는지]
   - 의심되는 근본 원인: [가설 1~2개]
   - 권장 조치: [사용자가 결정해야 할 선택지]
   ```

**Anti-pattern**: 같은 수정을 3회 이상 시도하거나, 테스트를 단순히 삭제/비활성화해서 "통과"시키는 것은 금지.

## 자동화 권장

- 모든 파일 편집 후 자동으로 이 검증을 실행하세요.
- `/commit-push-pr` 실행 전에 반드시 `/verify-app`을 실행하세요.
- 2-Strike 발동 후 사용자 확인 없이 3차 수정 금지.
