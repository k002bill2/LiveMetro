### 예제 2: 버그 수정 워크플로우 (LiveMetro)

#### Scenario: 실시간 도착 정보 갱신 안 되는 버그

```bash
# Claude Code 시작
claude

# 버그 정보 제공
"실시간 도착 정보가 화면에서 갱신되지 않는 버그가 있어.
HomeScreen에서 도착 정보를 폴링하는데,
화면을 떠났다가 돌아오면 데이터가 멈춰 있어.
src/screens/HomeScreen.tsx의 useEffect를 확인해줘"
```

#### useEffect cleanup 누락 디버깅

```
You: "HomeScreen에서 화면 전환 후 도착 정보가 멈춰 있어"

Claude: useEffect를 분석해보겠습니다...

[파일 분석 결과]

문제를 찾았습니다. useEffect에서 setInterval로 폴링을 설정했지만
cleanup 함수에서 clearInterval을 호출하지 않고 있습니다.
화면을 떠날 때 이전 interval이 정리되지 않아
새 화면에서 상태 업데이트가 충돌합니다.

수정사항:
1. useEffect return에서 clearInterval 호출
2. React Navigation의 useFocusEffect 활용하여
   화면 포커스 시에만 폴링 활성화
3. 화면 언포커스 시 폴링 중단
```

#### 테스트 작성 및 수정

```bash
# 테스트 작성
"이 버그에 대한 회귀 테스트를 작성해줘.
- useEffect cleanup이 제대로 호출되는지 검증
- 화면 포커스/언포커스 시나리오 테스트
- Jest와 @testing-library/react-native 사용"

# 수정 검증
/verify-app
```

### 예제 3: 새 기능 개발 (전체 플로우) - 역 주변 정보 기능

#### 1. Planning Phase

```bash
# Claude와 기획 논의
"역 주변 정보 기능을 추가하려고 해.
선택한 역 근처의 편의시설(편의점, ATM, 화장실) 정보를 보여주는 기능이야.
- 사용자 위치 기반 가까운 역 탐색
- 역 선택 시 주변 편의시설 목록 표시
- Firebase에 시설 정보 저장
구현 계획을 세워줘"
```

#### 2. Architecture Design (Expo Location, Firebase)

```bash
# 아키텍처 설계
"역 주변 정보 기능의 아키텍처를 설계해줘.
- Expo Location으로 사용자 위치 획득
- Firebase Firestore에 시설 데이터 저장 (facilities 컬렉션)
- 데이터 흐름: Expo Location -> Firebase -> AsyncStorage (캐시)
- @hooks/useNearbyFacilities 커스텀 훅 설계
- @models/Facility.ts 타입 정의"
```

#### 3. Implementation (mobile-ui-specialist agent 활용)

```bash
# UI 구현 - mobile-ui-specialist agent 활용
"mobile-ui-specialist agent를 사용해서 다음을 구현해줘:
- NearbyFacilitiesScreen.tsx 화면 컴포넌트
- FacilityCard.tsx 시설 정보 카드 컴포넌트
- FacilityList.tsx 시설 목록 컴포넌트
모두 TypeScript strict mode, 경로 별칭 사용"

# 서비스 계층 구현
"@services/facilityService.ts를 만들어서
Firestore에서 시설 데이터를 조회하고 캐싱하는 로직을 구현해줘"
```

#### 4. Testing (test-automation-specialist agent)

```bash
# 테스트 작성 - test-automation-specialist agent 활용
"test-automation-specialist agent로 다음 테스트를 작성해줘:
- NearbyFacilitiesScreen 컴포넌트 테스트
- useNearbyFacilities 훅 테스트
- facilityService 서비스 테스트
- 커버리지 기준: 75% statements, 70% functions, 60% branches"

# 전체 검증
/verify-app

```

```bash

"프로젝트 전체에서 상대 경로 import를 경로 별칭(@/)으로 변환해야 해.
Agent Teams를 활용해서 병렬로 진행해줘.

에이전트 구성:
- mobile-ui-specialist (sonnet): src/screens/*.tsx, src/components/*.tsx 변환
- test-automation-specialist (haiku): __tests__/**/*.test.tsx 변환 및 검증
- quality-validator (haiku): 변환 후 타입 체크 및 빌드 검증

각 에이전트가 독립적으로 작업하고,
quality-validator가 최종 검증 수행"

/verify-app
```

#### 리팩토링 실행 과정

```
1. 분석 단계
   - 상대 경로 사용 파일 목록 추출
   - 의존성 그래프 분석으로 변환 순서 결정

2. 병렬 변환 단계
   - mobile-ui-specialist: UI 관련 파일 변환
   - test-automation-specialist: 테스트 파일 변환

3. 검증 단계
   - quality-validator: npx tsc --noEmit 실행
   - 전체 테스트 스위트 실행 (2071 tests)

4. 완료
   - /commit-push-pr로 커밋 및 PR 생성
```

## 일반적인 문제 해결

### Case 1: 테스트에서 "Found multiple elements with text" 오류

```bash
# 원인: '로그인' 같은 텍스트가 제목과 버튼에 동시에 존재
# 해결: getByText 대신 getByTestId 사용

# 컴포넌트에 testID 추가
<TouchableOpacity testID="login-button">
  <Text>로그인</Text>
</TouchableOpacity>

# 테스트에서 testID로 조회
const loginButton = getByTestId('login-button');
```

### Case 2: useAuth mock 불완전으로 테스트 실패

```bash
# 원인: AuthContextType의 모든 필드를 mock하지 않음
# 해결: 전체 필드 포함

const mockAuth = {
  user: null,
  firebaseUser: null,
  loading: false,
  signInAnonymously: jest.fn(),
  signInWithEmail: jest.fn(),
  signUpWithEmail: jest.fn(),
  signOut: jest.fn(),
  updateUserProfile: jest.fn(),
  resetPassword: jest.fn(),
  changePassword: jest.fn(),
};
```

### Case 3: 같은 수정을 3회 이상 반복 (2-Strike Rule)

```bash
# CLAUDE.md 규칙: 2회 실패 후 접근 전환
# 1회차 수정 -> 실패
# 2회차 수정 -> 실패
# 3회차: 멈추고 근본 원인 분석

# 올바른 접근:
"이 버그를 2번 수정했는데 계속 실패해.
근본 원인을 분석하고 다른 접근 방법을 제안해줘"
```
